## Feature 1: Encryption at Rest — Documentation & S3 SSE Config (~3 days)

**Priority:** High (aerospace/defense requirement)
**Type:** OSS (docs) + Enterprise (S3 SSE config)
**Status:** Not started
**Carried over from:** 26.03.1 (rescoped — disk-level approach per Feb 16 research)

### Goal

Follow industry standard: document OS/cloud-level disk encryption and add S3 SSE config headers. No application-level encryption — this is what InfluxDB, QuestDB, ClickHouse, and DuckDB all recommend.

### Research Summary (Feb 16, 2026)

No major time-series database implements application-level encryption at rest. The entire industry treats it as an infrastructure concern:
- **InfluxDB:** recommends LUKS/dm-crypt
- **QuestDB:** recommends OS/cloud-level encryption
- **ClickHouse:** recommends LUKS/dm-crypt + S3 SSE
- **TimescaleDB:** PostgreSQL TDE extensions
- **DuckDB:** defers to OS-level encryption

### Implementation

1. **Documentation:** Encryption at rest guide covering:
   - Local disk: LUKS (Linux), dm-crypt, FileVault (macOS), BitLocker (Windows)
   - S3: SSE-S3 (default AES-256), SSE-KMS (customer-managed keys)
   - MinIO: KES + Vault setup
   - Azure Blob: SSE with Microsoft-managed or customer-managed keys
   - Docker: encrypted volumes
   - Kubernetes: encrypted PVCs

2. **S3 SSE config headers:**
   ```toml
   [storage.s3]
   sse_algorithm = ""       # "AES256" or "aws:kms"
   sse_kms_key_id = ""      # KMS key ARN (only for aws:kms)
   ```
   Env vars: `ARC_STORAGE_S3_SSE_ALGORITHM`, `ARC_STORAGE_S3_SSE_KMS_KEY_ID`

3. **Pass headers on PutObject calls** in `internal/storage/s3.go`

### Acceptance Criteria

- [ ] Encryption at rest documentation published
- [ ] S3 SSE config headers added and passed on upload
- [ ] Existing S3 tests still pass
- [ ] Guide covers all supported storage backends

## Feature 2: Proxy Support (~4 days)

**Priority:** High (defense sector, locked-down environments)
**Type:** OSS
**Status:** Not started
**Carried over from:** 26.03.1

### Goal

Allow Arc to route all outbound HTTP/HTTPS traffic through a proxy. Required for S3/Azure access in air-gapped networks, license validation through corporate firewalls.

### Implementation

1. **Configuration:**
   ```toml
   [proxy]
   http_proxy = ""
   https_proxy = ""
   no_proxy = ""
   # Also respects HTTP_PROXY, HTTPS_PROXY, NO_PROXY env vars
   ```

2. **Proxy transport factory:** `internal/proxy/transport.go`
   - `NewTransport(cfg ProxyConfig) *http.Transport`
   - Supports HTTP, HTTPS, SOCKS5 proxy URLs
   - Falls back to environment variables if config is empty

3. **Integration points:**
   - S3 client (`internal/storage/s3.go`)
   - Azure client (`internal/storage/azure.go`)
   - License client (`internal/license/client.go`)

### Files to Create/Modify

- `internal/proxy/` — new package (transport.go, transport_test.go)
- `internal/config/config.go` — add ProxyConfig
- `internal/storage/s3.go` — inject proxy transport
- `internal/storage/azure.go` — inject proxy transport
- `internal/license/client.go` — inject proxy transport
- `cmd/arc/main.go` — wire proxy config

### Acceptance Criteria

- [ ] S3 storage operations work through HTTP/HTTPS proxy
- [ ] Azure storage operations work through proxy
- [ ] License validation works through proxy
- [ ] `no_proxy` bypass list respected
- [ ] Environment variables (`HTTP_PROXY`, etc.) work as fallback

## Feature 3: Msgpack v6 Release & Arc Integration (~2 days)

**Priority:** Medium
**Type:** OSS
**Status:** Done

### Goal

Release the Basekick-Labs msgpack fork as v6 with a proper module path, then update Arc to use it.

### Steps

1. Update fork's `go.mod` module path to `github.com/Basekick-Labs/msgpack/v6`
2. Tag release `v6.0.0` on the fork
3. Update Arc's `go.mod`: replace `github.com/vmihailenco/msgpack/v5` with `github.com/Basekick-Labs/msgpack/v6`
4. Update all import paths in Arc
5. Run full test suite + ingestion benchmarks to verify no regressions

### Results

- **+8.6% ingestion throughput**: 16.78M → 18.23M rec/s
- **-10% p50 latency**: 0.52ms → 0.47ms
- **Flatter degradation curve**: 13% drop over 60s vs 22% with upstream

### Acceptance Criteria

- [x] Msgpack fork tagged as v6.0.0 with proper module path
- [x] Arc uses the fork via proper `go.mod` dependency (no replace directive)
- [x] All Arc tests pass
- [x] Ingestion benchmarks show no regression — significant improvement

## Feature 4: Slow Query Logging (~2 days)

**Priority:** Medium
**Type:** OSS
**Status:** Done

### Goal

Add configurable slow query threshold with WARN-level logging and a Prometheus counter. Infrastructure already exists (execution timing, query registry, latency metrics).

### Implementation

1. **Configuration:**
   ```toml
   [query]
   slow_query_threshold_ms = 1000   # 0 = disabled
   ```

2. **Changes to query handler** (`internal/api/query.go`):
   - After query completes, check `executionTime > slowQueryThresholdMs`
   - Emit `Warn()` log with SQL, duration, row count, token name
   - Increment `slow_queries_total` Prometheus counter

3. **New metric** (`internal/metrics/metrics.go`):
   - `slowQueriesTotal` counter
   - `RecordSlowQuery()` function

### Files Modified

- `internal/config/config.go` — added `SlowQueryThresholdMs` to QueryConfig, default, and loading
- `internal/api/query.go` — added `logSlowQuery` helper, wired into 3 query completion paths
- `internal/api/query_arrow_json.go` — wired into Arrow JSON completion path
- `internal/metrics/metrics.go` — added `querySlowTotal` counter, `IncSlowQueries()`, Prometheus output
- `cmd/arc/main.go` — passed `SlowQueryThresholdMs` to `NewQueryHandler`
- `arc.toml` — added commented config example

### Acceptance Criteria

- [x] Queries exceeding threshold logged at WARN level
- [x] Log includes SQL, duration, row count, token identity
- [x] `arc_slow_queries_total` Prometheus counter exposed
- [x] Threshold of 0 disables slow query logging
- [x] Existing query tests still pass

## Feature 8: Migration Guides (~2 days)

**Priority:** Medium
**Type:** Docs
**Status:** Not started

### Goal

End-to-end migration guides for the three most common source databases. Documentation only, no code changes.

### Guides

1. **InfluxDB → Arc**
   - Export via `influx query --raw` or InfluxDB API
   - Import via `POST /api/v1/import/lp` (shipped in 26.03.1)
   - Schema mapping: InfluxDB measurement/tags/fields → Arc measurement/tags/fields
   - Continuous query migration
   - Retention policy mapping

2. **QuestDB → Arc**
   - Export via `COPY TO` CSV
   - Import via `POST /api/v1/import/csv`
   - SQL dialect differences
   - Timestamp handling

3. **TimescaleDB → Arc**
   - Export via `COPY TO` CSV or Parquet (via pg_parquet)
   - Import via `POST /api/v1/import/csv` or `POST /api/v1/import/parquet`
   - Hypertable → Arc measurement mapping
   - Continuous aggregate migration

### Acceptance Criteria

- [ ] InfluxDB migration guide with working examples
- [ ] QuestDB migration guide with working examples
- [ ] TimescaleDB migration guide with working examples
- [ ] Each guide covers: export, import, verify, schema differences
