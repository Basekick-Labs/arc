# Arc v2026.06.2 Release Notes

> **Status:** In progress (planned release: July 2026).

> **This is a patch release.** Bug fixes and hardening only — no new features, no breaking API changes, no schema migrations. Drop-in upgrade from v2026.06.1.

## Security hardening

This release strengthens internal verification routines on the Arc Enterprise activation path and tightens authorization on database-management API endpoints. The changes are internal to Arc; existing license keys, activation flows, tokens, and `arc.toml` configurations continue to work unchanged.

**`GET /api/v1/logs` now requires an admin token.** The application-logs endpoint was registered among the public base routes (health, readiness, metrics), which placed it ahead of the global authentication middleware in the router and left it reachable without a token when authentication was enabled. It now requires an admin token, matching every other management endpoint. The exposure was limited to operational log metadata — component names, source-file locations, log messages, and request outcomes; Arc does not write query text, database or measurement names, or credentials to this buffer, and the first-run admin-token banner is printed to stderr rather than through the logged buffer, so none of those were exposed. The JSON metrics endpoints under `/api/v1/metrics` remain reachable without a token by design — the same as the Prometheus `/metrics` endpoint, they expose only aggregate operational counters and runtime statistics — and are now explicitly marked public so their status is a deliberate, audited decision rather than a side effect of route ordering. Deployments running without authentication are unchanged. Thanks to **@sondt99** for reporting this.

**Clustering now requires a shared secret and fails closed if one isn't set.** Arc Enterprise clustering authenticates inter-node messages (join, heartbeat, leave, replication) with an HMAC derived from `cluster.shared_secret`. A node configured with `cluster.enabled = true` but no shared secret now refuses to start, with a clear message to set `ARC_CLUSTER_SHARED_SECRET` — previously such a node would start and skip inter-node authentication, which is unsafe on any network where the coordinator port is reachable by untrusted hosts. This makes the secret mandatory for clustering, the same way it was already mandatory for peer replication. As additional hardening, cluster heartbeat messages are now HMAC-authenticated alongside the join and leave messages that already were. The shared secret has been recommended in the clustering documentation since the feature shipped, so correctly-configured clusters continue to run unchanged; only a cluster intentionally run without a secret needs to set one before upgrading. Clustering is an Enterprise feature and is off by default, so single-node and OSS deployments are unaffected.

**Rolling-upgrade note:** the HMAC message format for join, leave, and heartbeat messages now includes a per-message-type label ("join" / "leave" / "heartbeat") as the first field of the signed payload (#504). An upgraded node signs and validates with this label; a not-yet-upgraded node uses the old format (no label). During the mixed-version window, join requests, leave notifications, and heartbeats all fail HMAC validation across version boundaries — an old node rejects a new node's labeled join/leave/heartbeat as "shared secret mismatch," and a new node rejects an old node's unlabeled join/leave (and treats its unsigned heartbeat as unauthenticated). This does not affect Raft membership or cause data loss — it only suppresses read routing to legacy nodes — and it clears automatically once all nodes are upgraded. **Upgrade all cluster nodes in one window** (lockstep); expect transient "node unhealthy" and "authentication failed" log lines during the roll.

Thanks to **@sondt99** for reporting this through coordinated disclosure.

**Query endpoints now consistently require read permission.** The main query routes (`/api/v1/query`, `/query/msgpack`, `/query/estimate`, `/measurements`, `/query/:measurement`, and the Arrow IPC endpoint) previously accepted any valid token — a token with only `write` permission could execute SELECT queries. They now require a token with `read` permission (or an admin token), matching the permission model the write endpoints already enforce for ingest. Deployments that use only the default admin token or that have `read,write` tokens are unaffected; deployments that issue single-permission tokens and rely on the query routes accepting write-only tokens should ensure those tokens carry the `read` permission. The RBAC layer continues to provide table-level authorization as before.

**Auth database file permissions tightened.** The SQLite database that stores API tokens and RBAC roles — together with the `-wal` and `-shm` files SQLite creates alongside it — is now explicitly locked to owner-only permissions (`0600`). Previously these were created with the process umask, which defaults to world-readable (`0644`) on most systems, so on a multi-user host another local user could read the token database (and the write-ahead log can hold recently-committed token hashes before checkpoint). The fix pre-creates the main database file with restricted permissions before SQLite initializes it, re-applies `0600` after initialization as a guard against pre-existing files from earlier versions, and applies `0600` to the WAL/SHM siblings. When the configured database path is a symlink, the resolved (real) target is used to locate the WAL/SHM files so they are hardened correctly; if symlink resolution fails the server logs a warning rather than silently skipping the step. Token values are never stored in plaintext — tokens are bcrypt-hashed — but tightening these file permissions is a defense-in-depth improvement.

**Backup and credential documentation.** The config backup path (`backupConfig`) and the cloud-credential accessors (`GetAccessKey`, `GetSecretKey`, `GetAccountKey`) now carry doc comments noting that they handle plaintext secrets and must be treated accordingly. No code behavior changes — these are documentation-only clarifications. Operators who back up `arc.toml` should ensure backup storage has the same access controls as the live configuration.

`POST /api/v1/databases` and `DELETE /api/v1/databases/:name` now consistently require admin-permission tokens (matching the established convention on every other mutating endpoint in Arc). Read-only routes — listing databases, listing measurements, fetching database metadata — remain accessible to any authenticated token.

**`POST /api/v1/query/estimate` and `GET /api/v1/measurements` now enforce RBAC read-permission checks.** These two endpoints were missing the table-level RBAC authorization that the main `POST /api/v1/query` endpoint already performs. `estimateQuery` now checks read permission for every table referenced in the submitted SQL, resolves bare table references against the `x-arc-database` header (preventing a header-based RBAC bypass), and rejects cross-database syntax (`db.table`) when the header is set — matching the validation in `executeQuery`. The permission check normalises the SQL — masking string literals, masking `FROM` keywords inside function bodies (e.g. `EXTRACT(YEAR FROM time)`), and stripping comments — before extracting table references, matching the normalisation the query-transform path already applies. This closes a bypass where a comment interleaved between a `FROM`/`JOIN` keyword and the table name (`FROM /* x */ secret.cpu`) could hide a reference from the check while still resolving to that table at execution. It also avoids false-positive denials: a field name following an in-function `FROM` (`EXTRACT(EPOCH FROM ts)`), a CTE name (`WITH t AS (…) … FROM t`), or a table-valued function (`FROM generate_series(…)`) is no longer mistaken for a real table — the extractor now skips CTE names and function calls, matching the query-transform path. The cross-database-syntax guard applies the same normalisation (including in-function `FROM` masking), so legitimate queries such as `EXTRACT(EPOCH FROM t.ts) FROM cpu t` are no longer rejected when the `x-arc-database` header is set. **`SHOW DATABASES` and `SHOW TABLES`/`SHOW MEASUREMENTS` commands are now separately gated on the estimate endpoint:** these commands contain no `FROM`/`JOIN` table references, so the general `checkQueryPermissions` extraction path would pass them through unchecked. The estimate endpoint now mirrors `executeQuery`'s permission checks — `SHOW DATABASES` requires read on `*.*`, `SHOW TABLES`/`SHOW MEASUREMENTS` requires read on the target database — and returns a `400` directing callers to use `/api/v1/query` instead, since the estimate endpoint has no legitimate use for metadata commands. The SHOW gate on every query endpoint (JSON, MsgPack, Arrow IPC, and estimate) now matches against comment-stripped SQL so a comment cannot hide a `SHOW` from the anchored pattern, recognises quoted database names (`SHOW TABLES FROM "my-db"`), resolves an unqualified `SHOW TABLES` against the `x-arc-database` header, and validates the resolved database name with the standard identifier check — rejecting path-traversal tokens such as `SHOW TABLES FROM ..` before any storage access (relevant when RBAC is disabled). Multi-statement queries are now rejected at validation across all query endpoints: a second statement smuggled behind a semicolon (`SHOW DATABASES; SELECT 1`) previously slipped past the anchored SHOW patterns; a single trailing `;` is still accepted, and semicolons inside string literals or comments do not trip the check. `listMeasurements` requires read permission scoped to the requested database filter (falling back to `*.*:read` when no filter is specified) — matching the existing SHOW DATABASES / SHOW MEASUREMENTS authorization pattern in the query endpoint — and validates the `database` query parameter before use. The RBAC check is skipped when RBAC is not configured (OSS deployments) or when the token has no RBAC memberships (token-level permissions apply). Overhead is negligible: 2 ns disabled / 52 ns allowed / 186 ns denied for a single permission check, 6–30 µs for SQL table-reference extraction (same cost already paid by `executeQuery`).

**User SQL can no longer call DuckDB's filesystem-I/O functions (completes the CVE-2026-47735 fix).** The 26.06.1 sandbox fix blocked the literal `read_parquet()` and `arc_partition_agg()` spellings, but DuckDB exposes the same reads through the `parquet_scan()` alias and a wider family of path-taking table functions (`glob`, `read_blob`, `read_csv`/`read_csv_auto`, `read_json*`, `read_ndjson*`, `read_text`, `parquet_metadata`/`parquet_schema`, `read_xlsx`, `delta_scan`, `iceberg_scan`, and related). Because the DuckDB sandbox allowlists the entire storage root and the RBAC layer does not authorize function calls, any authenticated principal — including a least-privileged RBAC-scoped token — could read another database's Parquet data, file listing, and column statistics within the deployment by naming one of these functions directly (e.g. `SELECT * FROM parquet_scan('…/other-db/…')`). The query validator now rejects this entire family by name, in any query position — including comma cross-joins (`FROM a, parquet_scan(…)`), subqueries, IN-lists, lateral joins, and quoted-identifier spellings (`"parquet_scan"(…)`, which DuckDB executes identically to the unquoted call) — across every query endpoint (JSON, MsgPack, Arrow IPC, and estimate). Only Arc's own query-transform layer, which runs after authorization and carries the caller's identity, emits `read_parquet`; legitimate user SQL never needs these functions. Pure, non-I/O table functions (`generate_series`, `range`, `unnest`, …) and all scalar/aggregate functions are unaffected. (RBAC-enabled multi-tenant deployments were the exposure; single-database deployments were not cross-tenant-affected.)

**Tiering router `read_parquet()` builders now escape interpolated paths (#307, defense-in-depth).** The `internal/tiering` router's `read_parquet()`-building helpers interpolated file paths into DuckDB SQL without escaping single quotes. These helpers are not on the live query path — the query engine builds tiered `read_parquet()` calls through its own already-escaped `quotePath` helper — so this is hardening of currently-unused builders rather than a live-exposure fix: any future caller that wires them up inherits the escaping, and a regression test pins the contract. Enterprise-only (tiering).

**S3 credentials are no longer readable through the query API.** Arc previously configured DuckDB's S3 access by setting the access key and secret key as global DuckDB settings (`SET GLOBAL s3_secret_access_key=…`). DuckDB exposes settings to plain SQL, so any authenticated principal could read the secret key back with `SELECT current_setting('s3_secret_access_key')`. S3 credentials are now stored in DuckDB's secrets manager via `CREATE SECRET` (TYPE S3) — the same mechanism Arc already used for Azure — which is not reachable through `current_setting()` and is redacted in `duckdb_secrets()` (the non-secret access key ID remains visible there, the secret value does not). The secret is created in-memory (not persisted to disk) and is shared across the connection pool. Primary storage and the cold tier now use **separate, scope-bound secrets** (`arc_s3_primary` scoped to the primary bucket/prefix, `arc_s3_cold` scoped to the cold bucket/prefix), so a deployment that uses different buckets or credentials for hot and cold S3 has each tier authenticate with its own credentials — DuckDB resolves the right secret per `read_parquet()` path. (Previously, hot and cold shared a single global credential set, so configuring the cold tier overwrote the primary credentials.) This affects only deployments that configure S3 (or S3-compatible, e.g. MinIO) storage. As part of the change, S3 access now supports the AWS credential chain: when S3-compatible storage is configured with no static access/secret key, Arc provisions its secret with `PROVIDER CREDENTIAL_CHAIN`, so IAM instance roles, IRSA, and environment credentials work for queries — for both the cold tier and primary storage (see the primary-storage credential-chain fix under Bug fixes). A half-configured credential pair (exactly one of access key / secret key set) is rejected: for primary storage this fails startup; for a cold tier it is logged and that tier's queries fail, rather than silently authenticating as a different identity. Azure Blob Storage gains the matching credential-chain behavior: configuring only an account name (no account key) now provisions an Azure secret with `PROVIDER CREDENTIAL_CHAIN`, enabling managed identity / `az login` / environment credentials (previously this path was unreachable and an account key was effectively required). Azure also gets the same separate-scoped-secret treatment as S3 — primary and cold-tier Azure use distinct, container-scoped secrets (`azure_secret_primary`, `azure_secret_cold`), and DuckDB can now query a cold-tier Azure container with its own credentials (previously DuckDB only ever held the primary Azure secret, so a cold tier with a different container/account was not queryable). Found during verification of an external audit report.

**Continuous-query and parallel-executor `read_parquet()` paths now escape interpolated single quotes (defense-in-depth).** Two `read_parquet('<path>')` call sites — the continuous-query aggregation runner and the parallel partition executor — interpolated storage paths into DuckDB SQL without doubling embedded single quotes, the same escaping the rest of the query engine already applies (DuckDB does not support parameterized `read_parquet()`). The paths derive from validated database and measurement names, so the practical exposure was narrow, but both sites now escape consistently with the documented convention.

**RBAC-only API tokens can now be created with no platform permissions.** `POST /api/v1/auth/tokens` with `"permissions": []` is documented to create a token whose access comes solely from its RBAC team/role grants. Previously an empty permission set was silently upgraded to the default `read,write` (wildcard on every database), so the least-privilege token could not actually be created through the API — a privilege-escalation-shaped gap. An explicit empty array now persists as no platform permissions; omitting the field (or sending `null`) still applies the `read,write` default as before.

Operators on 26.06.1 should plan to upgrade.

## Supply-chain hardening

Arc now ships machine-readable software bill of materials (SBOM) and signed vulnerability scan reports as first-class release artifacts, targeting EO 14028 and defense supply-chain review requirements.

**SBOM generation.** Every release now includes two SBOM files generated by [Syft](https://github.com/anchore/syft):

- `arc-VERSION-sbom-container.spdx.json` — SPDX JSON generated from the published container image; captures the DuckDB native libraries, SQLite, and all Debian packages in the runtime layer.
- `arc-VERSION-sbom-source.cyclonedx.json` — CycloneDX JSON generated from the Go source tree; names every Go module with its exact version and SPDX license expression.

Both files are attached to the GitHub release and can be ingested directly by DCSA/CMMC supply-chain tooling. Both formats are provided because toolchain expectations vary across prime contractors.

**Vulnerability scanning.** Every release container image is scanned with [Trivy](https://github.com/aquasecurity/trivy) (CRITICAL/HIGH/MEDIUM findings) and the Go module graph is scanned with [`govulncheck`](https://pkg.go.dev/golang.org/x/vuln/cmd/govulncheck) (Go's official vulnerability database). Scan outputs:

- `arc-VERSION-trivy-report.json` — full Trivy JSON report for the container image, attached to the GitHub release.
- SARIF results are uploaded to the GitHub Security tab for every release.
- `govulncheck` runs on both build architectures (amd64 and arm64) in CI; a finding blocks the binary build.

The deliverable is signed scan evidence attached to each release, not "zero findings" — transitive OS CVEs from the Debian base layer outside Arc's control are reported but do not gate the release.

**Signed releases and SLSA Level 3 provenance.** Every release binary and container image is cryptographically signed using [Sigstore/cosign](https://github.com/sigstore/cosign) with keyless OIDC signing — no key material is stored or managed; the GitHub Actions OIDC token is the identity, anchored in the [Rekor](https://rekor.sigstore.dev) public transparency log. Air-gapped and Zarf-based deployments can verify artifacts offline using the bundled signature files.

- `arc-linux-amd64.bundle`, `arc-linux-arm64.bundle` — cosign signature bundles for each binary (signature + Rekor transparency-log entry). Verify with: `cosign verify-blob arc-linux-amd64 --bundle arc-linux-amd64.bundle --certificate-identity-regexp "^https://github.com/Basekick-Labs/arc/" --certificate-oidc-issuer https://token.actions.githubusercontent.com`
- Container images on GHCR are signed by manifest digest. Verify with: `cosign verify ghcr.io/basekick-labs/arc:VERSION --certificate-identity-regexp "^https://github.com/Basekick-Labs/arc/" --certificate-oidc-issuer https://token.actions.githubusercontent.com`
- `arc-VERSION.intoto.jsonl` — [SLSA Level 3](https://slsa.dev/spec/v1.0/levels) provenance attestation for the release binaries, generated by the [slsa-github-generator](https://github.com/slsa-framework/slsa-github-generator). Proves who built the artifact, from what source commit, and with what build inputs. Verify with: `slsa-verifier verify-artifact arc-linux-amd64 --provenance-path arc-VERSION.intoto.jsonl --source-uri github.com/Basekick-Labs/arc`

**FIPS 140-3 build variant (`arc-fips`).** This release adds an optional FIPS build of Arc for US defense/aerospace and other regulated environments that require validated cryptography. The FIPS variant is the **same source at the same version** as the standard build — it is not a separate product or version line. It is compiled with a `fips` build tag against the **CMVP-certified Go Cryptographic Module v1.0.0** (`GOFIPS140=v1.0.0`) and runs the module in FIPS-only mode (`GODEBUG=fips140=only` is baked into the binary). It carries the same Arrow/DuckDB performance build (`duckdb_arrow`) as the standard build.

- **Artifacts:** `arc-fips-linux-amd64`, `arc-fips-linux-arm64` (each with a cosign `.bundle`); container images tagged `:VERSION-fips` on GHCR and Docker Hub; `arc-fips` `.deb` and `.rpm` packages (which `Conflicts`/`Provides`/`Replaces` the standard `arc` package — install one or the other; both ship `/usr/bin/arc` + `arc.service`); plus a FIPS container SBOM (`arc-VERSION-fips-sbom-container.spdx.json`) and FIPS Trivy report (`arc-VERSION-fips-trivy-report.json`). The standard `arc-linux-*` artifacts, `:VERSION` images, and `arc` packages are unchanged. SLSA provenance covers all four binaries (standard + FIPS). Both variants report the same version (e.g. `26.06.2`); the FIPS variant is identified by its artifact name and by `"fips_mode":true` in its startup log, not by a different version string.
- **What runs through the FIPS module:** TLS for the API, cluster, and MQTT paths is restricted to FIPS-approved cipher suites and curves; API-token hashing uses **PBKDF2-HMAC-SHA256** (replacing bcrypt, which is not FIPS-approved); cluster replication key derivation uses the standard-library `crypto/hkdf`. The FIPS binary links **no** non-approved crypto (verified in CI by an import-graph check) and **fails closed at startup** if it is not actually running in FIPS mode.
- **Token-hash verification is bounded against abuse:** the PBKDF2 verifier rejects an oversized or malformed hash with a cheap length check *before* decoding, caps the iteration count, and pins salt/key lengths — so a corrupted or hostile stored hash cannot drive a CPU/memory denial of service at auth time. Cluster token replication applies the same length caps on both the create and rotate paths, so a rogue node cannot persist an oversized hash into peers' state. These guards apply to **both** build variants.
- **Token rotation on cutover (required):** API tokens created by a non-FIPS build are stored as bcrypt hashes, which the FIPS build **refuses to verify** (it denies the request and logs the reason at debug level). When moving a deployment to the FIPS build, **rotate (recreate) existing API tokens** so they are re-stored as PBKDF2 hashes. New tokens are PBKDF2 automatically.
- **Crypto boundary:** all cryptography is provided by the Go Cryptographic Module. DuckDB and SQLite perform no cryptography (SQLite is used for token *storage*, not encryption) and are outside the module boundary.
- **CMVP status (read carefully):** Arc's FIPS build is compiled against the CMVP-**certified** Go Cryptographic Module v1.0.0. This means the *cryptographic module* is validated; Arc itself is **not** a CMVP-listed module, and this is **not** a claim that "Arc is FIPS 140-3 validated." Confirm the live certificate number on the [NIST CMVP list](https://csrc.nist.gov/projects/cryptographic-module-validation-program) before relying on it in a compliance submission.

Verify the FIPS binary: `cosign verify-blob arc-fips-linux-amd64 --bundle arc-fips-linux-amd64.bundle --certificate-identity-regexp "^https://github.com/Basekick-Labs/arc/" --certificate-oidc-issuer https://token.actions.githubusercontent.com`. Verify the FIPS image: `cosign verify ghcr.io/basekick-labs/arc:VERSION-fips --certificate-identity-regexp "^https://github.com/Basekick-Labs/arc/" --certificate-oidc-issuer https://token.actions.githubusercontent.com`.

Arc Enterprise customers get FIPS by running the `arc-fips` build with their license key — there is no separate enterprise FIPS binary.

## Bug fixes

**Primary S3 storage now authenticates query reads via the AWS credential chain (IRSA / IAM role / env), not just static keys.** When `storage.backend` is `s3` or `minio` with no static access/secret key configured (the standard setup for EKS deployments using IRSA — IAM Roles for Service Accounts), Arc previously created no DuckDB S3 secret at all, so `s3://` query reads failed with an opaque "Loading external extensions is disabled" / permission error even though the pod's IAM role was valid and writes to S3 succeeded. The DuckDB query path now provisions an `arc_s3_primary` secret with `PROVIDER CREDENTIAL_CHAIN` whenever the primary backend is S3-compatible, scoped to the primary bucket/prefix so it coexists with a cold-tier secret. Deployments that configure static keys are unchanged (they still get a `KEY_ID`/`SECRET` secret), and a half-configured pair (exactly one key set) still fails startup. The Azure equivalent is also fixed: an Azure backend configured with a connection string (`storage.azure_connection_string` / `tiered_storage.cold.azure_connection_string`) but no account name now provisions its DuckDB secret from that connection string — for both primary and cold-tier Azure — so `azure://` query reads authenticate (previously the DuckDB secret was account-name-only, so a connection-string-only Azure deployment could write but not query). As a related guard, an S3 primary backend without `storage.s3_bucket` — or an Azure primary backend without `storage.azure_container`, or without either `storage.azure_account_name` or `storage.azure_connection_string` — now fails fast at startup with a clear message instead of producing an unscoped/missing secret and confusing query-time errors, and an unrecognized `storage.backend` value is rejected at load rather than deeper in startup. The same startup validation also covers the cold tier: when `tiered_storage.enabled` and `tiered_storage.cold.enabled` are both true, an invalid `backend` or a missing bucket/container/account is rejected at load rather than failing later during tiering or a multi-tier query. (Cold-tier config is not validated when the parent `tiered_storage.enabled` is false, matching the runtime, which ignores the cold tier entirely in that case.)

**Pre-epoch (pre-1970) timestamps now partition into the correct hour (#312).** The ingest hour-bucketing used plain integer division (`time / microsecondsPerHour`), which truncates toward zero rather than flooring. A negative timestamp — a pre-1970 date, which Line Protocol and MessagePack both accept — was therefore filed one hour too late: a row at `1969-12-31 23:30` landed in the `1970/01/01/00/` partition instead of `1969/12/31/23/`, so a time-range query for the pre-1970 hour would miss it. Hour bucketing now floors toward negative infinity, so a timestamp always partitions into the hour that actually contains it; non-negative timestamps are unaffected (floor and truncation agree). The in-process CSV/Parquet import `partitions_created` count uses the same corrected bucketing. This fixes go-forward writes; any pre-1970 data already written by an affected build would need to be re-ingested or recompacted to move into the correct partition.

**Compaction no longer wedges a partition when files disagree on the `time` column type.** A regression introduced in 26.05.1 (#411) made the ingest buffer's schema-change signature *type-aware* (`name:type`). As a side effect, when a writer sent the `time` column as a non-integer type for some batches (e.g. a client encoding timestamps as strings) while other batches sent integer epochs, the differing signatures routed those batches into **separate Parquet files** within the same hour/day partition. Compaction reads a partition with `read_parquet(union_by_name=true)`, which reconciles column *names* but not conflicting *types*, so the `time` column failed to bind (`INTERNAL Error: TIMESTAMP WITH TIME ZONE != VARCHAR`) and the partition could never compact — failing on every cycle while raw files accumulated. This is fixed at three levels:

- **The `time` column is forced to an integer-microsecond timestamp at ingest.** Every typed write coerces `time` to `int64` (→ Arrow `Timestamp`, stored as `TIMESTAMP WITH TIME ZONE`); a `time` column sent as a **string** (e.g. an ISO-8601 datetime) or containing **null** values is now rejected with a clear error at ingest rather than written as a `VARCHAR` (or misrouted to the 1970 partition). Because `time` can no longer vary in type across batches, it can no longer fan out into type-conflicting files — which is the actual cause of the wedge. (The schema-change signature stays type-aware, which is required to keep *other* columns whose type legitimately changes across batches from being merged into a single buffer and panicking; only `time` is constrained.)
- **Compaction is now resilient to already-written mixed-type partitions.** The compaction query normalizes `time` to `TIMESTAMP WITH TIME ZONE` (`COALESCE(TRY_CAST("time" AS TIMESTAMPTZ), make_timestamptz(TRY_CAST("time" AS BIGINT)))`), matching the timezone-aware type Arc writes for normally-ingested data. Partitions that already contain a mix of `TIMESTAMP WITH TIME ZONE` and `VARCHAR` `time` files from an affected 26.05.1 deployment will compact successfully on the next cycle with no manual intervention. Normalizing to a *naive* `TIMESTAMP` was avoided because it would strip the timezone and re-introduce the same type mismatch (compacted-naive vs newly-ingested-tz) on the following cycle. The dedup compaction path (measurements with tag metadata, including the daily tier) materializes this normalization before the de-duplication window so the dedup key sees the normalized `time` — avoiding both a DuckDB plan-binding error and a subtler silent under-dedup where the window would otherwise group rows by the raw, un-normalized timestamp. (See the follow-up fix below: the original CTE-based materialization in this release did not hold at the scale and schema width of real partitions, and was replaced with a temp-table materialization.)

**Dedup compaction no longer wedges on wide, many-file partitions (follow-up to the `time`-type fix above).** The dedup compaction path (measurements with tag metadata) wrapped the `time` normalization in a CTE that the de-duplication window read from. On real partitions — many files and the full column width of a typical measurement (e.g. `host, time, value, cpu_idle, cpu_user`) — DuckDB does not treat that CTE as a materialization barrier: the window's `time` reference is resolved against the raw `read_parquet(union_by_name=true)` scan, which fails to bind with `INTERNAL Error: Failed to bind column reference "time": TIMESTAMP WITH TIME ZONE != VARCHAR` — **even when every file's `time` is already a valid `TIMESTAMP WITH TIME ZONE` and no `VARCHAR` file is present anywhere** (the `VARCHAR` in the message is a binder artifact, not a real column type). The earlier release's test used a narrow three-column fixture that did not reach the binder's column-count threshold, so the failure surfaced only on production-shape partitions, where affected measurements failed to compact on every cycle and accumulated raw files. The dedup query now materializes the normalized read into a temporary table (`CREATE OR REPLACE TEMP TABLE … AS SELECT * REPLACE(<time normalize>) FROM read_parquet(…)`) and runs the de-duplication window over that table, so the window binds against a concrete `TIMESTAMP WITH TIME ZONE` column. The two statements are pinned to a single connection (a DuckDB temp table is connection-local) and the temp table is dropped before the connection is released. The standard (no-tag) compaction path is unchanged. Deduplication semantics (last-write-wins on `(tags, time)`) are unchanged.

**CSV and Parquet bulk imports now work against the shipped DuckDB version.** `POST /api/v1/import/csv` and `/api/v1/import/parquet` previously introspected the uploaded file by running DuckDB queries against it (`read_csv` / `read_parquet` + `DESCRIBE`); the `DESCRIBE`-as-subquery form was rejected by the DuckDB version Arc ships, so CSV imports failed with an HTTP 422 parser error before any data landed. Both formats now parse rows in-process and ingest through the same streaming pipeline as Line Protocol and TLE imports — no DuckDB queries against the uploaded file.

User-visible behavior changes:

- A 0-byte or header-only upload now returns `400 "file is empty"` / `"file contains no rows"` (previously a confusing parser error).
- Imports validate up front and reject, with a clear `400`: empty, blank, or duplicate column names; a `time_column` rename that would collide with an existing `time` column; or (for Parquet) a `NaN`/`Inf` value in a floating-point time column — cases that could previously have silently dropped a column, crashed the import, or written corrupt partition boundaries.
- Time columns now accept **fractional (floating-point) epochs** (e.g. `1609459200.123`), preserving sub-second precision; the old DuckDB path tolerated these and the in-process rewrite restores parity. Integer epochs keep full precision (no float round-trip). For Parquet, the time column may be a `TIMESTAMP`, an integer or floating-point epoch, or a timestamp string.
- CSV uploads are now subject to the same 500 MB size cap already enforced for the other import formats.
- Parquet `DECIMAL` columns are imported as `DOUBLE` (Arc's ingest path does not carry per-column decimal precision for imports). Use Line Protocol with a configured decimal column if exact decimal precision is required.

Performance: CSV imports now parse each numeric value once (the type-inference and conversion passes were merged), roughly halving CPU on numeric-heavy files; per-column buffers are pre-sized from the upload size to avoid repeated reallocation on large files. No behavior change.

Line Protocol and TLE imports are unchanged. CSV/Parquet imports no longer depend on the DuckDB sandbox's allowed-directories list.

**Line Protocol parser: eliminated per-value `strings.ToLower` allocation.** The field-value parser (`parseFieldValue`) was calling `strings.ToLower` on every ingested field value — even for floats and integers where the lowercase result was immediately discarded. Boolean detection now uses a zero-allocation byte comparison, saving one allocation per field value on the ingest hot path. Real-world throughput: ~5.3M rec/s (up from 4.6M), p50 latency 1.81ms (down from 2.14ms).

**Compaction metric fields are no longer exported.** `Manager` (`TotalJobsCompleted`, `TotalJobsFailed`, `TotalFilesCompacted`, `TotalBytesSaved`, `TotalManifestsRecov`) and `BaseTier` (`TotalCompactions`, `TotalFilesCompacted`, `TotalBytesSaved`) previously had exported struct fields that could be read or written without going through the mutex-protected `Stats()` / `GetBaseStats()` accessors, creating a potential data race. The fields are now unexported; the `Stats()` and `GetBaseStats()` methods remain the sole access path and continue to hold the mutex. `BaseTier` also gains a thread-safe `RecordCompaction(filesCompacted int, bytesSaved int64)` helper for tier implementations to increment metrics under the lock.

**MQTT subscription error handling now uses sentinel errors.** The MQTT API handler previously compared error strings for flow control (`err.Error() == "subscription already running"`, etc.) — fragile against message changes and re-wrapping. Four sentinel errors are now defined in the `mqtt` package (`ErrSubscriptionAlreadyRunning`, `ErrSubscriptionNotRunning`, `ErrSubscriptionRunningCantUpdate`, `ErrSubscriptionUniqueConstraint`), the manager returns them directly, and the API handler uses `errors.Is`. The `isUniqueConstraintError` helper in the repository was also updated to recognize the new sentinel alongside the existing SQLite string check. No user-visible behavior change.

**S3 and Azure listing loops now check context cancellation.** `List`, `ListDirectories`, and `ListObjects` on both `S3Backend` and `AzureBlobBackend` previously did not check `ctx.Err()` between paginated API calls. On very large prefixes (many pages), a cancelled or timed-out context would not propagate until the next SDK call, leaving the loop running longer than necessary. Every pagination loop now checks context cancellation at the top of each iteration, returning immediately when the context is done.

**Tiering policy lookups for databases without a custom policy no longer hit SQLite on every call (#345).** The tiering policy cache only stored found policies, despite a code comment claiming otherwise — so every lookup for a database using the global defaults (the common case) queried SQLite, twice per file per migration cycle. Not-found results are now cached alongside found ones; creating or deleting a policy invalidates the entry exactly as before, and a concurrent policy creation racing a lookup can no longer be shadowed by a stale not-found entry. Enterprise-only (tiering). One edge-case behavior change: a policy row written to SQLite outside the API (manual edit, restore-from-backup) was previously picked up by the next uncached lookup; with not-found results cached, it now requires a restart (or a `PUT` through the API) to become visible.

**A tiering policy lookup racing a policy deletion can no longer re-cache the deleted policy (#499).** A lookup that read the policy row just before `DELETE /api/v1/tiering/policies/:database` removed it could commit that stale result to the in-memory cache after the deletion — keeping the deleted policy (e.g. a `hot_only` exclusion) in effect on that node until the next policy write or a restart. The cache now carries a generation counter bumped by every policy write; a lookup whose read was overtaken leaves its result uncached, and the next lookup re-queries. Pre-existing edge case surfaced by the #345 review; enterprise-only (tiering).

**Tiered storage migration history now has periodic cleanup.** The `tier_migrations` metadata table previously grew without bound — every migration attempt (successful or failed) was recorded and never cleaned up. Each migration cycle now deletes records older than `[tiered_storage].migration_history_retention_days` (default: 90 days). OSS deployments are unaffected (tiering requires an enterprise license).

**WAL reader now uses `io.ReadFull` for fixed-size header reads.** The WAL reader previously used `f.Read` to read fixed-size file and entry headers. `f.Read` may return fewer bytes than the buffer size without an error, which could cause partial header reads that cascade into misaligned subsequent reads — corrupting all entries after the partial read. Both header reads now use `io.ReadFull`, which guarantees the buffer is filled completely or returns an error.

**WAL writer now tracks write failures and attempts rotation.** Previously, `w.currentFile.Write()` errors were logged but not tracked (no counter for operators to monitor), and the same bad file handle kept being used for subsequent writes, causing cascading failures. Write errors now increment a `FailedWrites` counter (exposed in `Stats()` and as `arc_wal_failed_writes_total` in Prometheus), and a rotation to a new WAL file is attempted immediately. If the rotation succeeds, the entry is retried on the new file. Sync errors from `w.currentFile.Sync()` were also silently ignored and are now logged.

**S3 and Azure storage backends now track storage metrics (#349).** The local filesystem backend has always recorded storage read/write/error counters, but the S3/MinIO and Azure Blob backends recorded nothing — on cloud-storage deployments these counters sat at zero. Both backends now record the same counters on the same operations the local backend instruments: writes (including S3 multipart uploads), reads, streaming reads, and ranged reads, plus the error counter on their failure paths. In addition, `arc_storage_reads_total` and `arc_storage_read_bytes_total` are now exported on the Prometheus endpoint — they were previously tracked (and visible in the JSON metrics snapshot) but missing from the Prometheus text output, alongside the already-exported `arc_storage_writes_total`, `arc_storage_write_bytes_total`, and `arc_storage_errors_total`. One caveat inherent to streaming uploads: when a write is streamed with an unknown size (S3 multipart for unsized streams, Azure `UploadStream`), the write is counted but no bytes are added to the byte counter. For reads that fail mid-transfer (streaming or buffered), the bytes transferred before the failure are still added to the read-byte counter (on all backends, including local) — partial transfers are real network egress — while the operation counter only counts completed reads. An operation that fails because the caller cancelled or timed out its context (client disconnect, shutdown) does not increment the error counter — those are caller-side lifecycle events, not storage failures, and counting them would page operators for normal client behavior. This guard covers both wrapped context errors and plain network errors (broken pipe / connection reset) raised when a disconnecting client kills a streaming copy; it applies to every cloud-backend error path and to the local backend's streaming-read paths, whose destination writer can be a network connection. List, delete, stat, and existence checks remain uninstrumented on all backends, matching the local backend.

**Registering the post-compaction cache-invalidation callback is now synchronized with running compaction jobs (#351).** `SetOnCompactionComplete` wrote the callback field without holding the manager's mutex, while a completing compaction job read the same field concurrently — a data race under the Go memory model. The window is narrow (the callback is wired once at startup, shortly after the compaction schedulers start), and no incident has been attributed to it; the setter and the job-completion read now synchronize on the manager mutex. No behavior or configuration change.

**Compaction subprocess log lines are now forwarded at their original severity.** The compaction manager runs each job in a subprocess and re-emits the subprocess's stderr into the main log for visibility. Every line was previously forwarded at `INFO`, so a subprocess error (for example a DuckDB bind failure) appeared in the parent log at `INF` while its embedded JSON still read `"level":"error"` — confusing when scanning logs by severity. The forwarder now parses each structured log line's `level` field and re-emits it at the matching level, so subprocess errors surface as `ERR`. Lines that are not structured JSON (such as continuation lines of a multi-line stack trace) still fall back to `INFO`, and the original line content is preserved verbatim. Cosmetic only — no change to compaction behavior or to what is logged.

## Performance improvements

**Compaction cleanup now uses batch-delete APIs on S3 and Azure.** `deleteOldFiles` previously called `StorageBackend.Delete` once per compacted source file — on large compaction cycles (hundreds of files), this produced hundreds of sequential S3/Azure API calls. Both cloud backends already implemented `DeleteBatch` (S3 `DeleteObjects`, Azure `BlobBatch`) but the compaction cleanup path never used it. The path now prefers `BatchDeleter.DeleteBatch`, falling back to per-file `Delete` if the backend does not support batching. This reduces S3 DELETE API calls by up to 1000× and Azure calls by up to 256× on large compactions, with a proportional drop in cleanup-phase latency. Local-storage deployments are unaffected (a per-file loop is the correct implementation for a local filesystem).

Additionally, the Azure `DeleteBatch` implementation was rewritten to use the actual Azure SDK `BlobBatch` API (`container.Client.NewBatchBuilder()` + `SubmitBatch`) instead of a per-file loop, bringing it into parity with the S3 implementation.

**Paginated FSM manifest walks replace full-snapshot copies.** `ClusterFSM.GetAllFiles()` previously allocated a full O(N) copy of the file manifest — for 1M+ files this caused ~50ms apply-path spikes (RLock held while copying) and ~50MB transient memory. The replication catch-up walker and the `/api/v1/cluster/files` endpoint now use cursor-based pagination via `GetFilesPaginated(cursor, limit)`, which:

- Maintains a lazily-built sorted-key cache (invalidated on manifest mutation)
- Returns pages of entries, releasing the RLock between pages so Raft apply-path latency is unaffected by long-running walks
- Supports `?cursor=<path>&limit=<n>` query parameters on the API endpoint (backward-compatible — omitting both params returns all files as before)

The emergency kill-switch `replication_catchup_enabled=false` remains available.

**Ingest flush no longer builds-and-discards a per-hour index on the common single-hour path.** Every flush called `groupByHour`, which allocates an index bucket covering every buffered row, then discarded the result whenever the data fell within a single hour — the common case for live ingest (timestamps near "now"). On a 5M-row buffer this was ~201 MB allocated and immediately thrown away per flush. The flush now does a cheap inline min/max scan to make the single-vs-multi-hour decision and only builds the per-hour buckets on the multi-hour (backfill / clock-skew) path. The single-hour decision drops from ~16 ms / 201 MB to ~1.25 ms / 0 B per flush, cutting GC pressure under sustained load. No behavior change; multi-hour partitioning is unaffected.

**Ingest flush time-sort replaced with a radix sort.** The time-only sort that orders each flush before writing Parquet used a closure-based `sort.Slice`. Under concurrent producers the merged buffer is effectively unordered (profiling measured ~2.5M sorted runs in a 5M-row buffer), so this cost ~6.9% of ingest CPU. Because the sort key is a signed `int64` microsecond timestamp, an LSD radix sort with a sign-flip bias (so pre-1970/negative timestamps order correctly — see #312) replaces it, skipping the near-constant high-order byte passes — ~5× faster on the merged-buffer shape (796 ms → 157 ms on 5M rows), dropping the sort to ~2.5% of CPU. In-order single-producer ingest still short-circuits to an identity permutation (no sort, no copy). The radix sort is stable, so rows with equal timestamps are now written in deterministic arrival order (the previous `sort.Slice` left their relative order unspecified) — a benign improvement, not a behavior change callers can depend on either way. Combined with the single-hour flush change above, sustained MessagePack-columnar ingest improved from ~20.0M to ~20.9M rec/s over a 60-second run with a flatter throughput-decay curve and lower p99/p999 latency on the benchmark machine.

## Upgrade notes

1. **No configuration change required.** Drop in the new binary; existing `arc.toml` and license keys work as-is.
2. **Active licenses keep working.** Arc binaries running against `enterprise.basekick.net` continue to operate normally; no re-activation or license-key reissuance is required.
3. **OSS-only deployments** (no `[license]` block in `arc.toml`) are unaffected by the license-verification change. The database-API authorization change applies only when authentication is enabled in `arc.toml`.
4. **Token review for operators using non-admin tokens for database management**: if any automation provisions databases via `POST /api/v1/databases`, ensure the token it uses has the `admin` permission. Auto-create-on-write (databases that come into existence as a side-effect of line-protocol or msgpack writes) is unaffected — write tokens continue to work for ingestion.
5. **CSV/Parquet import callers**: imports that relied on a Parquet `DECIMAL` column round-tripping as a decimal will now receive a `DOUBLE`. Malformed files now fail fast with a `400` instead of partially succeeding (empty file, empty/blank/duplicate column names, a `time_column` rename that collides with an existing `time` column, or a `NaN`/`Inf` float time value) — review any automation that ignored import error responses.

Go 1.26.4 incorporates security fixes across `crypto/x509`, `crypto/tls`, `html/template`, `net/http`, `net/textproto`, `net/url`, and `os` (CVEs from Go patch series 1.26.1–1.26.4); `golang.org/x/net` v0.55.0 fixes GO-2026-5026 (Punycode label rejection in `idna`); `golang.org/x/crypto` v0.53.0 clears a batch of advisories reported against v0.51.0.
